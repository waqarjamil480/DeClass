06First principles

What is a database, really?

Strip everything away — a database is just three things:

  1. A place to put data so it survives a restart
  2. A way to find what you put there, fast
  3. Some guarantees about what happens when many people read & write at once

Different databases make different trade-offs on those three things:

DatabaseOptimised for
SQL (Postgres, MySQL)Strict structure, strong guarantees, complex queries
NoSQL (MongoDB, Redis…)Flexibility, scale, speed — accept some trade-offs to get them
"NoSQL" is not a single thing. It's a label for "the family of databases that aren't strict tables." We'll meet four kinds today.
07The "so what?"

Why a data engineer needs to know NoSQL

You will run into NoSQL databases constantly in real data engineering work:

  • Source systems — modern apps store user data in MongoDB / DynamoDB; you'll be the one extracting it
  • Event ingestion — clickstreams and IoT data land in document stores before being processed
  • Caching layers — Redis sits in front of almost every web service you'll ever touch
  • Serving layer — when analysts need sub-100ms lookups, the warehouse is too slow; NoSQL is the answer
Modern data systems are "polyglot" — they use multiple databases together. A data engineer who only knows SQL is missing half the toolbox.

04Why NoSQL exists

A scenario from 2006

You're an engineer at a brand-new social media company.

Users post photos, tag friends, comment, like, share. All this data lives in one big SQL database (think MySQL or PostgreSQL — the kind you've already met).

  • Month 1: 100,000 users → one server, no problem.
  • Month 6: 50 million users → server is on fire. 🔥
What do you do? The next slide shows what people actually tried — and why each option hurt.
05Why NoSQL exists

What people actually tried

StrategyWhat it meansWhat breaks
Buy a bigger box"Vertical scaling" — beefier CPU, more RAMHits a hardware ceiling, costs explode
Add read replicasCopy the DB to extra machines for readsReads OK, but all writes still hit one machine
Manual shardingSplit data across N servers (e.g. by user-id)App has to track which server holds what; joins across servers stop working
DenormalizePre-join data into wide tables to avoid JOINsSchema balloons; updates become multi-day events

Every workaround chips away at something SQL was designed to give you — strict structure, joins, transactional safety.

That's the crack NoSQL grew out of. A new generation of databases said: "what if we drop some of those guarantees on purpose, to get scale and flexibility back?"
06Why NoSQL exists

A real timeline

  • 2006 Google publishes the BigTable paper
  • 2007 Amazon publishes the Dynamo paper
  • 2008 Cassandra released by Facebook
  • 2009 MongoDB 1.0 · Redis 1.0
  • 2009 Term "NoSQL" coined at a meetup in San Francisco
NoSQL = "Not Only SQL" — not "No SQL". The movement isn't anti-relational. It's: there are workloads SQL doesn't fit. Modern practice = polyglot persistence.

08Axis 1 of 4

Axis 1: Schema

SQL

ALTER TABLE users ADD COLUMN tags VARCHAR(255);
-- can lock 500M rows

The DB refuses anything that doesn't match.

MongoDB

  name: "Grace",
  email: "g@x.com",
  tags: ["admin"]
});

Both docs in the same collection. No migration.

09Axis 1 of 4

Schema: the double-edged sword

UpsideDownside
Evolve without downtime"Schema-less" can become "schema-everywhere-in-the-code"

MongoDB's middle ground: schema validation rules (collection-level JSON schema). You opt-in to enforcement when you want it.
10Axis 2 of 4

Axis 2: Scaling

Vertical = bigger machine. Horizontal = more machines.

Why is going horizontal hard for SQL?

SQL promises that a transaction is all-or-nothing. If your data is spread across 8 machines, every multi-row transaction now has to coordinate across all of them and agree to commit or roll back together. That coordination is slow and easy to break when networks hiccup.

NoSQL relaxes some of those guarantees on purpose so it can spread across many machines without that coordination cost.

Same MongoDB API runs on:

  • Your laptop (one node)
  • A 1,000-node cluster across multiple data centres
Data engineer's lens: when you see "petabyte-scale" or "millions of writes per second" in a job description, this is the property that makes it possible.
11Axis 3 of 4

Axis 3: Data shape — impedance mismatch

The application has:

user = {
  "name": "Ada",
  "tags": ["admin", "beta"],
  "address": { "city": "London" }
}
StorageWhat it looks like
SQL3 tables (users, user_tags, addresses) joined via FKs
MongoDBLiterally that exact object, one document
In plain English: SQL forces you to break your nice nested object into flat tables, then JOIN them back together every time you read. MongoDB just stores the object as-is. (That's why an entire category of tools called "ORMs" — Object-Relational Mappers, like SQLAlchemy in Python — exists: to translate between objects and tables. With a document DB, you don't need that translation layer.)
12Axis 4 of 4

Axis 4: Consistency — preview

We'll do this properly in the next slides (CAP theorem). For now: MongoDB lets you tune consistency per query.

  • Want strong reads? Read from primary.
  • Want fast reads, OK with stale data? Read from secondary.
  • Want durable writes? w: "majority".
  • Want fastest writes? w: 1.
This dial doesn't exist in classical SQL.
13Distributed systems

The CAP Theorem

The most important idea in distributed databases

Brewer (2000) · Proven by Gilbert & Lynch (2002)

14CAP theorem

CAP — the statement

"Distributed" just means the database runs on more than one machine. CAP says such a system can guarantee at most two of these three things:

  • C — Consistency — everyone always sees the latest data
  • A — Availability — the database always answers (never down)
  • P — Partition tolerance — it keeps working even if some machines can't talk to each other
In plain English: when you have many machines holding copies of the same data, and the network breaks between them, you must choose: stop answering until you can sync up (consistency), or keep answering with possibly-stale data (availability).
C / \ / \ / CP \ /________\ A P \ AP / \ / \___/ Pick any two (in theory)
15CAP theorem

CAP — the catch

Network partitions WILL happen. Cables get cut. Switches fail. Cloud zones go dark.

P is not optional.

The real choice is: C vs A, during a partition.

16CAP theorem

CP vs AP

Choose C (CP)Choose A (AP)
Behaviour during partitionMinority side stops servingAll nodes keep serving
RiskUnavailabilityStale / divergent data
ExamplesMongoDB, HBase, Zookeeper, etcd, Redis (cluster)Cassandra, DynamoDB, CouchDB, Riak

Neither is "right" — depends on whether stale reads are worse than unavailability for your use case.

17CAP theorem

CP vs AP — concrete example

Replica set across two data centres. The link between them is cut. A customer in DC-A updates their shipping address. A customer in DC-B tries to read it.
SystemBehaviour
CP (MongoDB)DC-B returns error / "temporarily unavailable"
AP (Cassandra)DC-B returns the old address — package ships to wrong place

Correctness vs availability. Pick your poison based on the workload.

18Beyond CAP

PACELC — the extension

CAP only talks about partitions, which are rare. What about normal operation?

PACELC (Daniel Abadi):
if Partition → A or C
else → Latency or Consistency

In MongoDB under normal ops:

  • Read preference → primary (slow, fresh) vs secondary (fast, possibly stale)
  • Write concern → w: 1 (fast) vs w: "majority" (durable)

You're tuning the L-vs-C dial every time you query.

19Consistency models

ACID — what SQL gives you

ACID is the safety promise SQL databases make about transactions. (A "transaction" = a group of operations that should succeed or fail as one unit — like moving money between accounts.)

  • Atomicity — all-or-nothing. Either every step succeeds, or none do.
  • Consistency — the database moves from one valid state to another (rules like "no negative balances" hold)
  • Isolation — two transactions running at the same time don't see each other's half-finished work
  • Durability — once it says "done", it survives a crash, power cut, anything
The classic example: moving $100 from account A to account B. ACID guarantees you can never lose the money or duplicate it, even if the server crashes mid-transfer.
20Consistency models

BASE — the original NoSQL pitch

Early NoSQL systems traded ACID for a more relaxed model called BASE:

  • Basically Available — the database always answers, even if it's degraded
  • Soft state — the data may change over time even without new writes (because replicas are catching up)
  • Eventually consistent — if you stop writing, all copies will agree… eventually (milliseconds usually)
Real-world example: when you "like" a post on Instagram, the count might be slightly different on two phones for a second. That's eventual consistency in action — the system chose speed over instant agreement.
21Consistency models

Where MongoDB actually sits

  • Single-document writes have always been atomic. A document is the unit of atomicity.
  • MongoDB 4.0 (2018) — multi-document ACID transactions across a replica set
  • MongoDB 4.2 — multi-document ACID across a sharded cluster
MongoDB is not a "BASE database" anymore. It's a CP system with ACID at the document level and optional ACID across documents.
22Modeling principle

The modeling rule that follows

"Embed what you read together."

Because single-doc writes are atomic, the art of MongoDB modeling is shaping documents so that things you update together live in the same document.

Do that, and you rarely need transactions.

23Landscape

The four NoSQL families

One slide each · one concrete product · one sweet-spot use case
24Family 1 of 4

1. Document stores

Examples MongoDB, CouchDB

Shape: self-describing JSON-like documents in collections.

Query: rich — filter by any field, including nested + array fields.

Sweet spot: user profiles, product catalogs, CMS, content — anything where the aggregate ("everything I need to render this page") is the natural unit.

This is us today.
25Family 2 of 4

2. Key-value stores

Examples Redis, Memcached, DynamoDB

Shape: opaque value addressed by a single key.

Query: by key only. O(1).

Sweet spot: caches, session storage, rate limiters, leaderboards, pub/sub. "I know the key — give me the value RIGHT NOW."

Mental model: a giant Python dict that survives restarts.
26Family 3 of 4

3. Wide-column stores

Examples Cassandra, HBase, Google BigTable

Shape: rows identified by a row key; each row has flexible columns grouped into column families.

Query: by row key + column ranges. Bad at arbitrary filtering.

Sweet spot: massive time-series (IoT, log events), write-heavy workloads, when you know your access pattern in advance.

Mental model: a spreadsheet where every row has different columns and there are 10 billion rows.
27Family 4 of 4

4. Graph databases

Examples Neo4j, Amazon Neptune, ArangoDB

Shape: nodes (entities) + edges (relationships), both with properties.

Query: traversal languages — Cypher, Gremlin. "Friends of friends who like jazz and live in Lagos."

Sweet spot: social networks, fraud detection, recommendations, knowledge graphs.

Mental model: if your whiteboard diagram has lots of arrows, you want this.
28Polyglot persistence

None of these replaces SQL

  • A bank's core ledger → still PostgreSQL / Oracle (you need ACID for money)
  • Netflix's viewing history → Cassandra (massive write volume)
  • Your user profile service → MongoDB (flexible shape per user)
  • Their session cache → Redis (sub-millisecond lookups)
  • Airbnb uses all of these together — that's polyglot persistence.
Use the right store for the right job.
Data engineer's takeaway: in a real job you won't pick "SQL or NoSQL" — you'll move data between them. Knowing what each is good at is how you design the pipeline.

31Mental model

BSON — Binary JSON

Documents on disk and on the wire are BSON, not JSON.

Adds types JSON doesn't have:

  • ObjectId
  • Date (ISODate)
  • NumberLong, NumberDecimal
  • Binary, Regex

Other useful properties: length-encoded → fast random field access without parsing the whole doc · field order preserved.

Numbers aren't "just numbers." {count: 1} might be Int32, Int64, or Double.
32Mental model

The ObjectId — 12 bytes

| 4 bytes: timestamp | 5 bytes: random | 3 bytes: counter |

Properties:

  • Globally unique without coordination (no auto-increment bottleneck)
  • Roughly time-ordered → newer IDs sort later (great for pagination)
  • Extract creation time: 
    ObjectId("507f1f77bcf86cd799439011").getTimestamp()
// ISODate("2012-10-17T20:46:22Z")
33Mental model

The 16 MB document limit

16 MB

Every document is hard-capped at 16 MB.

If you find yourself designing a doc that grows unboundedly:

  • All comments embedded in the post doc
  • All GPS pings embedded in the trip doc

…you have a modeling problem.

The fix: keep the unbounded thing in its own collection and reference it.

35Pre-load these

Two gotchas to memorise now

1. address.coord = [longitude, latitude]
Lng first. GeoJSON convention. Everyone says "lat/lng"; computers store "lng/lat." This disagreement eats hours.
2. grades is an array of sub-documents
When we query "grades.score": {$gt: 70}, MongoDB matches if any element of the array satisfies the condition.

Internalise these two. They explain half of today's "weird" results.

39The query API

Projection rules

1 = include · 0 = exclude

You cannot mix include and exclude in one projection — except _id, which can always be excluded.

_id is returned by default — opt out with _id: 0.

// LEGAL
{ name: 1, borough: 1, _id: 0 }

// ILLEGAL — mixing include + exclude
{ name: 1, address: 0 }

45Nested fields

Dot notation — nested fields

{ "address.zipcode": "10075" }
{ "address.coord": [...] }

Wrap the dotted path in quotes — JS would otherwise treat the dot as object access.

46Array semantics

Querying arrays — the key idea

A condition on an array field matches if ANY element satisfies it.
// grades = [ {grade:"A", score:10}, {grade:"B", score:80} ]
db.restaurants.find({ "grades.score": { $gt: 70 } })
// ✅ matches — because score 80 > 70

This is the most important behaviour to internalise for today's lab.

47Array semantics

Positional access in arrays

"address.coord.0"   // longitude (first element)
"address.coord.1"   // latitude  (second element)

Use this when you want a specific element, not "any element."

52Lab query 4

Q4 — Two valid forms

// Form 1 — classic answer
// Matches if ANY element of coord < -95.754168
// (in this data, that's longitude — lat is never that negative)
db.restaurants.find({ "address.coord": { $lt: -95.754168 } })

// Form 2 — explicit by index
db.restaurants.find({ "address.coord.0": { $lt: -95.754168 } })
Trust the data, not the prose.
53Lab query 5

Q5 — Multiple conditions (with $and)

db.restaurants.find({
  $and: [
    { cuisine: { $ne: "American " } },
    { "grades.score": { $gt: 70 } },
    { "address.coord": { $lt: -65.754168 } }
  ]
})

Three conditions joined explicitly with $and.

54Lab query 6

Q6 — Same query, no $and

db.restaurants.find({
  cuisine: { $ne: "American " },
  "grades.score": { $gt: 70 },
  "address.coord": { $lt: -65.754168 }
})

Identical results. Multiple keys → implicit AND.

Implicit AND is idiomatic Mongo. Use it.
55Lab query 7

Q7 — Filter + Projection + Sort

db.restaurants.find(

58Regex

MongoDB regex — two equivalent syntaxes

// Native regex literal — works in mongosh
{ name: /^Wil/ }

// String form — works in any driver (Python, Node, Java, …)
{ name: { $regex: "^Wil" } }

// With flags
{ name: { $regex: "^wil", $options: "i" } }   // case-insensitive

Regex = "regular expressions" — a tiny pattern language for matching text. MongoDB uses the same flavour as Python and JavaScript, so anything you've seen there works here.

59Regex

Anchors — the only three you need today

PatternMeaning
^Wilstarts with "Wil"
ces$ends with "ces"
Regcontains "Reg" anywhere

Combine with character classes for power: /^[AEIOU]/i = "starts with vowel."

60Regex

Performance note

Unanchored or case-insensitive regex cannot use a normal index efficiently.
  • Fine for labs (25k docs)
  • Costly on a 100M-doc collection in production

Rule of thumb: anchor your regex (^...) when possible, and avoid the i flag at scale.

Why ObjectId(id) matters

In the shell, ObjectId is implicit — strings get converted automatically.

In Python, you must wrap the string:

from bson.objectid import ObjectId

mongo.db.task.find_one({"_id": ObjectId("507f1f77bcf86cd799439011")})

If the string isn't a valid 24-char hex → bson.errors.InvalidId.

The Flask app catches this and returns HTTP 400.

73Pattern

Pattern: existence check before update/delete

existed_task = mongo.db.task.find_one({"_id": ObjectId(id)})
if existed_task is not None:
    mongo.db.task.update_one(
        {"_id": ObjectId(id)},
        {"$set": {"title": new_title}}
    )
    return jsonify({"message": "updated"}), 204
else:
    return jsonify({"error": "not found"}), 404

Realistic API shape: never just blindly update — always tell the client whether their target existed.
